Intro

目标

Goal:（一致性和完整性）
1.检查是否满足需求
2.检查是否有错误

但是测试不能证明程序是无错的

测试流程：

测试案例->测试数据->测试结果->测试报告

测试组织：

单元测试->集成测试（Integration testing）->功能性测试（完成的服务）->性能测试（和性能相关）->验收测试（Acceptance test）(用户执行)->Installation teset(安装测试)

验收测试：

有两种：Alpha测试与Beta测试：如果一个软件是给很多客户使用的，可使用Alpha测试与Beta测试。
Alpha测试：是在一个受控的环境下，由用户在开发者的指导下进行测试，由开发者负责记录错误和使用中出现的问题。

Beta测试：由最终用户在自己的场所进行，开发发者通常不在场，也不能控制应用的环境。由用户记录错误和使用中出现的问题，并定期地交给开发者来解决。

Component testing（unit testing）

测试内容：单个函数，单个类，单个部件

接口测试

背景：
1.Many components are made up of several interacting objects
2.Testing these composite components then is primarily concerned with testing that the component interface behaves

接口测试的一些一般准则：
1.检查要测试的代码，明确列出对外部组件的每次调用
2.在接口间传递指针时，始终使用空指针参数测试接口
3.在通过过程接口调用组件时，设计应导致组件失败的测试
4.在消息传递系统中使用压力测试，如上一节所述
5.当多个组件通过共享内存交互时，设计能改变这些组件激活顺序的测试

Integration Test

特点
1.系统集成过程包括从组件中构建一个系统，并测试由此产生的系统是否存在因组件相互作用而产生的问题。
2.集成测试过程中出现的一个主要问题是定位错误
3.为了更容易定位错误，应始终采用增量方法进行系统集成和测试
4.一个好的经验法则是先集成实现最常用功能的组件。这意味着使用最多的组件要接受最多的测试

注意：测试AB需要案例T1 T2 T3，集成ABC需要案例T1 T2 T3 T4 需要再运行一遍T1 T2 T3，避免C的影响

测试策略：
第一种 big bang：一次性集成所有模块
　不容易定位错误

第二种：逐步增加测试模块
　Bottom-up集成
　
　Top-Down集成（从A->ABCD->ABCDEFG）（注意每一层，在之前单独的进行测试）

Black-box and White-box testing

黑盒：不理解代码，不了解内部，根据程序的程序功能测试
　release testing

白盒：了解内部结构

Release testing（Functional testing）

检查功能满不满足用户的需求，不了解结构

找容易发生错误的案例进行测试

一边选择的案例：
1.Choose inputs that force the system to generate all error messages
2.Design inputs that cause input buffers to overflow
3.Repeat the same input or series of inputs numerous times
4.Force invalid outputs to be generated
5.Force computation results to be too large or too small

Performance testing

发现缺陷的有效方法是围绕系统的极限设计测试。在性能测试中，这意味着对系统施加压力–因此称为压力测试
Stress Testing（超出预计的限额，载重负载的情况下是否依然正常工作）

压力测试的两个功能：
1.测试系统失败的行为
2.给系统施压，可能造成通常不会出现的缺陷

Test case Design

三种方法：（不限于）
1.Requirements-based testing：根据需求测试（黑盒）

2.Partition testing：将输入输出划分为等价类，根据等价类选择对应的输入输出（黑盒）

3.Structural testing：根据代码结构设置（白盒）

需求测试

用户应能搜索所有初始数据库集或从中选择一个子集
系统应提供适当的阅读器，供用户阅读文件库中的文件
每个订单应分配一个唯一标识符（ORDER_ID），用户可将其复制到账户永久存储区

设置一个测试流程，覆盖多个情况

Partition testing

输入数据和输出结果通常划分为不同等价类

每一个类都是等价的部分，选择一个作为类的代表，表现出相同或者相似的运行结果和过程

测试案例从每一个部分选择

Structural testing

Structural testing is an approach to test case design where the tests are derived from knowledge of the software’s structure and implementation

Path testing

路径测试是一种结构测试策略，其目标是测试组件或程序的每一条独立执行路径。
基本路径法：白盒测试

特点：
1.通过程序的路径数量通常与程序的大小成正比
2.路径测试并不测试程序中所有路径的所有可能组合
3.路径测试的起点是程序流程图
4.路径测试的目的是确保程序中的每条独立路径都至少执行一次

什么时候停止测试

测试越多错误越少但是成本越高
什么时候停止测试是个问题->转化成：如何估计程序中的错误（两种办法）

跟踪开发过程中在每个组件中发现的故障数量，收集每个组件的测量信息（如大小、决策次数等）

方法：
分类树：一种统计技术，可对大量测量信息进行分类，并创建一棵决策树来显示最佳预测因子，
分类树有助于确定哪些组件可能会出现大量错误

Fault seeding（标志重捕）（估算错误）

把问题标注了 发给毫不知情的小组
Fault seeding：
标记了的错误/标记总错误=检测到的非标记/总的未标记

第二种方法（估算错误）

两个独立小组同时测试，
小组1发现x个错误
小组2发现y个错误
共同错误q个
估算所有的错误：n
n=xy/q
