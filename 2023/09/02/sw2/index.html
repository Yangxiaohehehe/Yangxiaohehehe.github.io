What is software

Program包含data以及Document
Def：把软件当成一种工业产品，用工程化的原理与方法对其进行开发和维护。总之就是经济有效的解决软件问题的一种工程。
Function：经济的获得可靠的且能在实际机器上高效运行的软件而建立和使用的完善工程原理

Software crisis

软件危机是指在计算机软件的开发和维护过程中所遇到的一系列严重问题。

  • 项目运行超出预算
  • 项目运行超过时间
  • 软件质量低落
  • 软件不匹配需求
  • 项目无法管理代码难以维护

CS和SE的relationship

Solving Problems（软件工程三要素）

  • Method
  • Tool
  • Procedure（过程）

Methods

Def:提供一系列软件开发技术。包括完成开发过程中各方面任务的方法并用某种特殊的语言或图形来描述

Tools

Def:软件工具为软件方法提供了自动的或半自动的支撑环境。将多种工具集成在一起可构成计算机辅助软件工程（ Computer-Aided Software Engineering 即CASE）的软件开发支撑系统。

Procedure

Def:a combination of tools and techniques to produce a product
软件过程是将软件工程的方法和工具综合起来，进行软件开发活动。

Software Lifecycle

包含开发 维护 使用

Software Process

Process Def：定义who is doing what， when and how达到一个目的。



具体过程来讲：

软件定义：可行性定义，需求分析

软件开发：概要设计，详细设计，实现，测试（集成测试，接受测试等），使用与维护

Software process model（软件过程模型）

Def:定义软件开发过程（包括活动，产生产品，人文等）的模型，是软件过程模型是软件过程的简化表示

model software development（开发过程模型）

  • Waterfall approach
  • Iterative development
  • Component-based software engineering（基于软件组成）
开发过程包括：
Analysis->Design->Implement->Tesing
在开发过程中有很多模型，我们在这里讨论最多的：Waterfall 和Iterative

Waterfall model（瀑布模型）

Def：是一个软件生命周期模型，开发过程是通过设计一系列阶段顺序展开的，从系统需求分析开始直到产品发布和维护，项目开发进程从一个阶段“流动”到下一个阶段，这也是瀑布模型名称的由来。


特点：一步一步做，朝向一个方向来，上一阶段完成再下一阶段

缺点：可能设计完了才会发现有问题（根据用户需求）

Iterative model（迭代模型）

在迭代模型中，软件在一系列步骤中被发展。不断地进行ADIT的步骤进行完善。


与瀑布模型不同点：在瀑布中是严格的串行，在迭代模型可以并行。把内容分为不同的小块(即增量)然后各自进行ADIT。

Analysis Phase

定义软件做什么而不规范软件如何被做

分类：根据实现阶段使用procedural 编程语言还是object-oriented语言分为两类单独的方法

POA

Procedure-Oriented analysis（面向过程）：即结构分析。是一种以过程为中心的编程思想。这些都是以什么正在发生为主要目标进行编程，不同于面向对象的是谁在受影响。与面向对象明显的不同就是封装、继承、类。简写为POP。

如果用这种语言实现，在Analysis阶段的规范会使用下列几种模型

Data flow diagram（数据流程图）

Def：show data在系统中的迁移

Entity-relationship diagram

ER流程图

State diagrams

when the state of the entities in the system will change in response to events then use.
就是entity的状态对应事件改变。

OOA

如果在实现阶段使用OOP的方法，在分析阶段可能会使用以下模型

Use case diagrams

show 用户如何和系统联系
四个组成：系统，用户案例，actor和relation

Class diagrams

类图描述系统静态结构。



在系统的逻辑视图中，类图用于表示类和它们之间的关系。我们利用类图来说明实体共同的角色和责任，这些实体提供了系统的行为。



类图中的两个基本元素是类和它们的基本关系。

Design Phase

Def:定义了系统如何实现设计阶段的内容，在设计阶段所有组成成分都被定义了

POD

面向过程设计：我们需要设计程序和数据，可以分为以下几个过程和模型

Structure charts

用来图解模型之间关系（在POD中）的工具

Modularity（模块性）

Def：单独解决模块可能比合在一起简单，简而言之就是分成小部分解决。


模块划分的好不好的度量指标：Coupling （耦合度，模块之间的联系程度）and cohesion（衔接）

Coupling(耦合度)

Def：耦合是软件模块之间的相互依赖程度


特点：模块之间的联系程度 之间越不相关越好，不用考虑直接的相互影响


三类：

  • Tightly coupled
  • Loosely coupled
  • Uncoupled（几乎不可能实现）
低耦合的优点：更易开发和维护
Cohesion(衔接)

Def：内聚是模块内元素组合在一起的程度（refers to the degree to which the elements inside a module belong together），是内聚是模块内元素组合在一起的程度。


有更高的衔接程度越完美，更可读，代码更易修改，测试更加容易

Difference

耦合是模块之间的相互依赖程度，耦合是关于如何改变一件事需要改变另一件事。

内聚是模块内元素的组合程度，简单地说，它的意思是“一起变化的代码，保持在一起”。

OOD

Def：设计阶段继续细化类的细节。类是由一组变量(属性)和一组方法组成的。面向对象的设计阶段列出了这些属性和方法的细节（感觉就是封装的具体内容）

Implement Phase

度量指标：satisfies the user’s requirements, meets the operating standards of the organization, and runs efficiently on the hardware for which it was developed


可操作性，可维护性，可移植性

Testing Phase

goal：find errors，好的测试就是找到最多的error


两种类型的测试：

  • glass-box (white box)
  • black-box

Glass-box

Def:基于软件内部代码，得到内部结构，测试是否components按设计运行。就是设计测试用例，看结果和预计的一不一样。

based on knowing the internal structure of the software,determine whether all components of the software do what they are designed for


分类：

  • Basis path
  • Control structure

Black-box testing

也称功能测试、数据驱动测试，它将被测软件看作一个打不开的黑盒，主要根据功能需求设计测试用例，进行测试。

也称功能测试或基于规格说明的测试。其基本观点是：任何程序都可以看作是从输入定义域到输出值域的映射，这种观点将被测程序看作一个打不开的黑盒，黑盒里面的内容(实现)是完全不知道的，只知道软件要做什么。

分类：

  • Exhaustive testing(穷举)
  • Random testing（随机）
  • Boundary-value testing（边界值）

Documentation

Def:为了软件能够正确的被使用以及高效的维护，需要文件，通常情况下，需要三个独立的问题：用户手册，系统手册和技术文档

User documentation

Def：也叫作user guide， 展示如何一步一步的使用软件，通常包含教程部分，指导用户了解软件的每个功能。

是非常厉害的营销手段

System documentation

Def：定义软件本身以至于可以被技术人员以外的人维护和修改，在软件开发的四个阶段应该都有

Technical documentation

Def:技术文档描述了软件系统的安装和维护（即有两部分）。安装文档定义了软件应该如何安装在每台计算机上，例如服务器和客户端。服务文档定义了系统应该如何维护和必要时如何更新。
