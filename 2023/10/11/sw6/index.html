INTRo

Def of requirement:指用户对软件的需求

Def of Requirement engineering：find out，分析，documenting，检查系统服务和限制的过程

涉及的人员：管理人饿元，分析人员，用户，软件开发小组

需求过程：elicitation（引出获取需求）->分析（理解并且建模）->Specification（规范）(软件的behavior)->validation（验证）->Software Requirement Specification(SRS)(需求规格文档)


Requirement和Specification的区别：

在环境域中的任何地方定义的需求，包括系统的接口

规范仅限于环境和系统域之间的交集

也就是说环境和系统的关系：




Types：

  • Functional requirement: 提供了功能性需求，描述软件需要完成哪些功能
    • 具有完整性，定义所有的需要的服务
    • 具有一致性，不应该相互矛盾
  • Non-functional requirement: 系统提供的服务或者功能的限制：包括时间限制，反应时间，可靠性等（不会直接影响某个功能），但如果不满足这些需求就是usable的（大致就是速度size和可靠性）
  • Domain requirements: 某些对应的领域的法规的限制

Software requirement：<br
User requirement：（概括）

是高度抽象的需求<br
用自然语言加上图表表示系统期望提供哪些服务以及系统必须在哪些约束下运行的陈述

对于不同的用户有不同的软件需求




System requirement：（详细）

细节的描述了系统需要做什么

细节的设置了系统的功能，服务和操作限制

应该be precise

PS：用户说的不一定很多都是功能，需要提取软件

Specification

Def：Requirement 用自然语言来写

规范的指导：

  • 发明一种标准格式
  • 始终如一地使用语言
  • 使用文本高亮来挑选需求的关键部分
  • 尽量避免使用计算机术语

自然语言的缺点：
  • 有些会容易引起误导
  • 自然语言的应用范围是某一概念内使用相同语言的读者和写作者
  • 自然语言书写格式灵活没限制
  • 没有一个容易的方式模块化自然语言需求

更多规划化表示：
    结构自然语言
  • 设计描述语言
  • 图表示
  • 数学规划范：

Structured language specifications

结构语言规范：

优点：保持自然语言的大部分表现力和可理解性；确保在规范上施加某种程度的一致性
